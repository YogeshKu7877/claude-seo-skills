**seo-log-analysis/SKILL.md** (LOCAL-02):
- Frontmatter: name `seo-log-analysis`, description covering server log crawl analysis, allowed-tools: `Read, Bash, Glob`
- Trigger words: "log analysis", "crawl budget", "server logs", "bot traffic", "crawl frequency", "access log"
- Inputs: `file` — absolute path to server log file (Apache Combined/Common or Nginx access log)
- Execution:
  1. Read first 10 lines to auto-detect format (Apache Combined: `%h %l %u %t "%r" %>s %b "%{Referer}i" "%{User-agent}i"`, Apache Common: no referer/UA, Nginx: similar with slight field differences)
  2. Parse log lines using Bash awk/grep (no Python dependency needed for basic parsing)
  3. Classify user-agents into categories: Googlebot, Bingbot, other search bots, AI crawlers (GPTBot, ClaudeBot, PerplexityBot, Bytespider), monitoring tools, real users
  4. Calculate: total requests, requests by bot category, requests by status code, top 20 crawled URLs by frequency, crawl frequency by path prefix (top 10 directories), requests by hour-of-day
  5. Identify crawl budget concerns: 4xx/5xx error rates, duplicate crawls, low-value path crawling
- Output Format: `## Server Log Analysis: [filename]` with sections for Crawl Budget Summary, Bot Traffic Breakdown (table), Top Crawled URLs (table), Crawl Frequency by Path (table), Status Code Distribution, Crawl Budget Recommendations

**seo-ai-content-check/SKILL.md** (LOCAL-03):
- Frontmatter: name `seo-ai-content-check`, description covering AI content detection, allowed-tools: `Read, Bash, Glob, WebFetch`
- Trigger words: "AI content check", "AI detection", "AI written", "content authenticity", "AI content", "detect AI"
- Inputs: `target` — URL (fetch via WebFetch) or file path (read directly). If URL, extract text content. If file, read directly.
- Execution (pure text analysis — no external API calls):
  1. **Sentence structure analysis**: Calculate variance in sentence length — AI text tends toward uniform sentence lengths. Flag if standard deviation of sentence length < 3 words (very uniform).
  2. **Vocabulary diversity**: Calculate type-token ratio (unique words / total words) for every 500-word window. AI text typically has lower diversity. Flag if TTR < 0.4 in any window.
  3. **Repetition patterns**: Detect repeated phrases (3+ word sequences appearing 3+ times). AI text over-uses transitional phrases. Flag commonly over-used AI phrases: "In conclusion", "It's worth noting", "It's important to note", "At the end of the day", "In today's [digital/fast-paced/modern] landscape", "Delve into", "Navigating the", "Leverage", "Cutting-edge", "Revolutionize", "Seamlessly", "Robust", "Comprehensive guide".
  4. **Hedging language**: Count hedging phrases per 1000 words: "may", "might", "could", "perhaps", "arguably", "generally speaking". Flag if hedging rate > 8 per 1000 words.
  5. **Paragraph uniformity**: Calculate variance in paragraph lengths. AI tends toward uniform paragraph sizes. Flag if standard deviation < 15 words.
  6. **Personal experience signals**: Detect first-person experience markers ("I tested", "we found", "in my experience", "our team"). Absence in opinion/review content is a signal.
  7. **Confidence score**: 0-100 scale. Weight: sentence uniformity 20%, vocabulary diversity 20%, repetition 20%, hedging 15%, paragraph uniformity 10%, experience signals 15%. Higher score = more likely AI-generated.
- Output Format: `## AI Content Analysis: [filename/URL]` with Confidence Score (0-100 with interpretation: 0-30 likely human, 30-60 mixed/edited, 60-100 likely AI), Detailed Findings per category, Flagged Passages (quote specific text with explanations), Recommendations for improving authenticity

**seo-internal-links/SKILL.md** (LOCAL-04):
- Frontmatter: name `seo-internal-links`, description covering internal link structure analysis, allowed-tools: `Read, Bash, Glob, WebFetch`
- Trigger words: "internal links", "link structure", "orphan pages", "internal linking", "link graph", "anchor text audit"
- Inputs: `domain` — target domain URL (e.g., https://example.com). Optional: `max_pages` (default 100, max 200)
- Execution:
  1. Crawl site starting from homepage using `scripts/fetch_page.py` + `scripts/parse_html.py`. Follow internal links only (same domain). Cap at max_pages. Respect robots.txt.
  2. Build link graph: for each page, record all internal links (source URL, target URL, anchor text).
  3. Calculate per-page metrics: inbound internal links count, outbound internal links count, link depth from homepage.
  4. Identify orphan pages: pages discovered via sitemap or other means but with 0 inbound internal links.
  5. Identify underlinked pages: pages with < 3 inbound internal links (these need more internal links pointing to them).
  6. For top 5 underlinked pages: suggest specific anchor text based on the page's H1/title and recommend which existing pages should link to them (pages that mention related topics).
  7. Identify excessive linking: pages with > 100 outbound internal links (dilutes PageRank).
  8. Detect broken internal links: links pointing to 404/5xx pages within the site.
  9. Optional Ahrefs enrichment: "If Ahrefs available (ToolSearch '+ahrefs'): fetch `site-explorer-pages-by-internal-links` to compare crawl-based findings with Ahrefs data. Add `### Ahrefs Internal Link Data` section."
- Output Format: `## Internal Link Audit: [domain]` with Summary Stats (total pages crawled, total internal links, avg links per page), Orphan Pages (table), Top 5 Underlinked Pages with Suggested Anchors, Broken Internal Links, Link Depth Distribution, Pages with Excessive Outbound Links, `## Data Sources` footer

**seo-local/SKILL.md** (LOCAL-05):
- Frontmatter: name `seo-local`, description covering local SEO audit, allowed-tools: `Read, Bash, Glob, WebFetch`
- Trigger words: "local SEO", "Google Business Profile", "GBP", "NAP consistency", "local citations", "local schema", "local pack", "map pack"
- Inputs: `business` — business name, URL, or location (e.g., "Acme Plumbing, Austin TX" or "https://acme-plumbing.com")
- Execution:
  1. If URL provided: fetch homepage using WebFetch, extract business info (name, address, phone from page content and schema markup)
  2. **NAP Consistency Check**: Extract Name, Address, Phone from homepage. Check if NAP appears consistently across: footer, contact page, about page, schema markup. Flag inconsistencies.
  3. **Local Schema Audit**: Check for LocalBusiness, PostalAddress, GeoCoordinates schema. Validate required properties. Generate missing schema as JSON-LD.
  4. **Google Business Profile Signals**: Check for: Google Maps embed on site, GBP link presence, reviews integration, service area mentions, operating hours on site.
  5. **Local Content Signals**: Check for: city/region mentions in title tags, location pages (if multi-location), local testimonials, service area content, proximity/location keywords.
  6. **Citation Opportunities**: List top 10 local citation sites relevant to detected business type (general: Yelp, BBB, Yellow Pages, Apple Maps; industry-specific based on detected type).
  7. **Review Signals**: Check for: review schema on site, links to review profiles, review aggregate scores.
  8. **Mobile Optimization**: Local searches are 60%+ mobile — check viewport meta, touch targets, click-to-call links, embedded maps.
- Output Format: `## Local SEO Audit: [business]` with Local SEO Score (0-100), NAP Consistency Status, Local Schema (present/missing with generated JSON-LD), GBP Integration Checklist, Local Content Assessment, Citation Opportunities (table), Priority Actions

**seo-migration-check/SKILL.md** (LOCAL-06):
- Frontmatter: name `seo-migration-check`, description covering site migration validation, allowed-tools: `Read, Bash, Glob, WebFetch`
- Trigger words: "migration check", "site migration", "redirect audit", "URL mapping", "domain change", "relaunch", "site move"
- Inputs: Either (a) two URLs: `old` and `new` for single-page check, or (b) a CSV/text file with URL pairs (one pair per line, comma-separated: `old_url,new_url`)
- Execution:
  1. For each URL pair:
     a. **Redirect Chain Validation**: Fetch old URL, follow redirects (max 5 hops). Check: redirect type (301 permanent vs 302 temporary — flag 302s as they don't pass full link equity), number of hops (flag > 2), final destination matches expected new URL.
     b. **Canonical Consistency**: Fetch new URL, check canonical tag matches the new URL (not pointing back to old URL or missing entirely).
     c. **Title/Meta Preservation**: Compare title tag and meta description between old (from redirect chain start) and new. Flag if significantly different (title changed by >50% of words, or meta description completely different). Warn — sometimes changes are intentional.
     d. **HTTP Status Codes**: Verify new URL returns 200. Flag 4xx/5xx.
     e. **Content Similarity**: If both old and new are fetchable, compare word counts. Flag if new page has <50% of old page's word count (possible content loss).
     f. **Schema Preservation**: Check if structured data types present on old URL exist on new URL.
  2. Summary statistics: total URL pairs checked, pass/fail/warn counts.
- Output Format: `## Migration Validation: [old] -> [new]` (or `## Migration Validation: [N] URL pairs`) with Pass/Fail/Warn summary table, Per-URL Results (table with columns: Old URL, New URL, Redirect, Canonical, Title, Meta, Status, Schema, Overall), Failed Items Detail (expanded info for any FAIL results), Recommendations
